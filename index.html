The problem

Winnable tenders are found too late, or not at all

Every public IT contract in the EU is announced in the open, on TED. In the last 30 days alone, 84,337 notices were published. The signal is public. The problem is scale and noise.

Published on TED
~2,700 notices/day

Peaking above 4,000 on busy days, across every sector and every member state.

Format
Deeply nested XML

eForms notices with multi-step reference chains. Not readable by humans at scale.

Today's process
Manual monitoring

Analysts sample the feed by hand. Coverage is partial and timing is reactive.

✕
No filter.IT relevant tenders (CPV 30, 48, 72) are buried in an all sector feed.
≈
No ranking.No systematic way to compare tenders on value, competition, and fit to Microsoft's portfolio.
◷
No early warning.Buyer intent signals in Prior Information Notices go undetected, so relationship building starts too late.
Our solution

One platform, four capabilities, zero manual steps

A production data platform on Databricks that ingests, understands, and ranks the entire EU IT procurement market, then serves it to analysts through a dashboard and a conversational copilot.

01

Automated pipeline

TED eForms XML lands in a Bronze, Silver, Gold medallion architecture. Parsed with lxml, modeled with dbt, stored in Delta.

02

Opportunity ranking

An XGBoost competition model feeds an expected value score for every open IT tender, every morning.

03

Buyer intelligence

An affinity score for every contracting authority, plus a PIN Monitor that surfaces buying intent before tenders open.

04

Procurement Copilot

An LLM agent that answers natural language questions, grounded exclusively in the Gold layer. It never writes SQL.

⚙️ Fully automated: a four task Databricks job (ingest, parse, dbt, ML scoring) runs on serverless compute every morning at 08:00 UTC. No manual steps anywhere in the pipeline.
Architecture

From raw XML to ranked opportunities, every morning

A classic medallion design, chosen because it keeps raw data immutable, transformations testable, and the serving layer fast.

Source

TED eForms XML

Daily packages from all EU member states. Raw files kept immutably, so any run can be replayed.

Bronze

Parsed notices

lxml extracts notices, lots, award criteria, and organizations into Delta tables. Checkpointed and incremental.

Silver

Clean, deduplicated lots

dbt models resolve re-published notices with two layer dedup and enrich lots with CPV reference data.

Gold

KPIs, IT lots, ML scores

Serving tables for the dashboard, the copilot tools, and the daily scoring job. One consistent source.

Users

Dashboard + Copilot

Seven page Streamlit app and a grounded LLM agent, both reading the same Gold tables.

✓Scoped to IT: CPV divisions 30 (hardware), 48 (software), and 72 (IT services), with four entities parsed per notice: notices, lots, award criteria, organizations.
✓Incremental and idempotent: each daily run parses only new files, checkpointed on source file. Raw XML is immutable, so parsing can be replayed at any time.
✓Award chain resolved: we walk the multi-step reference chain from lot to tender to winning supplier, and a curated CPV seed table labels every lot.
Design decisions

Every major choice was made against a considered alternative

Storage & compute
Chosen
Databricks + Delta + dbtThe most mature ecosystem for medallion, dbt, and ML in one place. The design ports to Fabric later.
Considered
Azure Synapse / MS FabricStrong strategic fit, but a less mature dbt and ML workflow for a five person capstone timeline.
Ranking model
Chosen
XGBoost classifierHandles high cardinality categoricals, non linear value effects, and missing data natively.
Considered
Logistic regression / SVMSimpler, but weaker on exactly the messy, mixed type features this data is full of.
User interface
Chosen
Streamlit + embedded copilotThe only option combining an analytical dashboard and a conversational agent in one Python app.
Considered
Power BIExcellent BI, but no natural home for a custom LLM agent inside the same experience.

Trade-offs were accepted knowingly. Streamlit gives up enterprise SSO and mobile layout, which are not critical for a focused analyst team.

Machine learning

From "number of bids" to an expected value ranking

The insight: tenders that attract few bidders are more accessible. Bid counts from award notices let us learn competition patterns with zero proprietary data.

Trained on real outcomes

The classifier predicts Low, Medium, or High competition from historical award notices: value, CPV, country, procedure, and buyer type.

Affinity that decays

Buyer affinity blends recency weighted CPV relevance (70%) with the share of open procedures (30%), on a 180 day half life. Recent behaviour counts double.

Honest by design

Scores are a systematic ranking, not calibrated win probabilities, and the dashboard says so. Microsoft's real bid history would calibrate it.

Critical analysis

What the system is not yet, and how it gets there

Known limitations
Proxy targetTrained on market wide bid counts, not Microsoft's win/loss history. It is a ranking, not a calibrated probability.
CoverageThe CPV division filter misses IT tenders filed under generic codes.
ValuesNon EUR lots pass through unconverted, and some lots lack values entirely, so headline sums are buyer reported and uncleaned.
Roadmap
Calibrate with real outcomesRecord Microsoft's bid decisions against scored tenders and retrain. The architecture already accommodates this.
NLP relevance scorerScore lot titles and descriptions to catch mis-classified tenders beyond CPV codes.
Scale upThe medallion design ports to Azure and Fabric when Microsoft standardises, with dbt models moving unchanged.

We would rather show you honest limits than inflated claims. Each limitation above has a designed path to resolution.
